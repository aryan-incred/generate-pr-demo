Phase-by-Phase — Decision Points & Retry Logic
Phase 0 · Story Selection
Branch detection + Resume
Reads domain-map.json first. Checks EPIC-REGISTRY for active stories only (not Shipped/Cancelled). Cross-checks user email against personas.md before showing any story.
🔁 Retry: PR has "changes requested" → enters revision mode → continues to Phase 1
Phase 0 · Hard Stops
3 conditions that exit immediately
① Email not in personas → stop, contact Tech Lead.
② Story has unresolved GAP-REPORT → stop, PM must fix story.
③ BUILD-EVIDENCE already on main → redirect to /mm-ship.
Phase 1 · Requirements Gate
@mm-scoping-analyst validates
If story was already validated (no GAP-REPORT on main), skips directly to Phase 1 Pass. Otherwise delegates to @mm-scoping-analyst with full story content — does not re-read files.
🔁 On FAIL: GAP-REPORT.md created, Slack message to PM, developer re-runs /generate-pr after fix
Phase 2 · Technical Blueprint
PLAN.md is the shared contract
@mm-codebase-planner reads SERVICE-KB.md from every service, traces call chains, resolves RESOLVE-IN-PLAN items, generates 3-section PLAN.md with Mermaid sequence diagram and p95 targets.
🔁 On Gate 1 rejection: @mm-codebase-planner revises — loops until [1] Approve. After approval, PR raised and waits for merge before Phase 3.
Phase 3 · TDD Red
Tests MUST fail — confirmed by running suite
@mm-test-architect writes all tests from PLAN.md Section 3. Tests are run immediately — if any pass, the pipeline stops with an investigation menu: feature already exists? test wrong? partial impl?
🔁 On unexpected pass: investigate root cause → revise tests → @mm-test-architect re-writes → re-run
Phase 4 · Function Impact Flag
Fires on existing function body change only
Checked before implementation starts. For every PARTIAL-REUSE function in PLAN.md, greps all domain services for callers. If callers found in other services: hard block until engineer confirms each one, or chooses to add a new function instead.
🔁 Stop for team discussion: developer exits, discusses with TL, then re-runs from Phase 4
Phase 4 · Implementation
Chunk-by-chunk with per-chunk gates
@mm-implementer only touches files listed in PLAN.md Section 2. After each chunk: runs tests, shows pass count, stops for approval. Gate 2 cannot be self-approved — requires a peer or Tech Lead.
🔁 Chunk rejected: revise → @mm-implementer retries same chunk. Gate 2 rejected: revise → full re-run of @mm-implementer
Phase 5 · Quality Gates
Hard block on any failure
Checks package.json before running lint or format — skips silently if script not registered. Runs: unit tests, integration tests, lint, format, coverage, p95 estimate, E2E tests. Any failure shows root cause and blocks.
🔁 Fix failure → re-run the specific failing gate (not all gates)
Phase 6 · PR Assembly
8-point sign-off before PR creation
PLAN.md ✓ · service bindings ✓ · sequence diagram ✓ · all tests green ✓ · no skips ✓ · impact flags reviewed ✓ · BUILD-EVIDENCE.md ✓ · QA-EVIDENCE.md ✓. Any red item → cannot proceed.
🔁 Red sign-off item: fix it, re-check sign-off list, then continue to Gate 3
Phase 7 · KB Update PR
Close the feedback loop
Creates a separate PR to monorepo main with BUILD-EVIDENCE, QA-EVIDENCE, EPIC-REGISTRY status update (→ In Pipeline), retrospective entry, and KB drafts marked "DRAFT — pending PM review". PM is notified on #mm-dev-pm.
After /generate-pr
Next: /mm-ship --qa
Once the implementation PR and KB PR are reviewed and merged, developer runs /mm-ship --qa to promote the story to the QA environment. @mm-qa-gatekeeper runs integration tests against live services and writes QA-EVIDENCE.md.
Error Handling — all cases
Every error state has a Slack message
Domain not registered → stop, explain how to add. Story GAP-REPORT → Slack to PM. Phase 2 PR not merged → Slack to PM + TL with PR link. No eligible approver → flag, add to personas. GitHub unavailable → print manual gh pr create command.
