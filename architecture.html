What each key controls at runtime
name
Display label in Phase 0 team-selection menu and in all gate messages
monorepo
Root of the PM monorepo — where PLAN.md, BUILD-EVIDENCE.md, QA-EVIDENCE.md are written
personas
Used in every phase — resolves who is PM, Tech Lead, VP, QA. Gates read this to assign reviewers and generate Slack pings automatically
epics_dir
Phase 0 scans this for active stories. Phase 1 reads the story file. Phase 2 writes PLAN.md here
registry
EPIC-REGISTRY.md — Phase 0 reads this to list active/in-progress epics. Prevents duplicate Epic IDs across concurrent team members
kb_dir
Knowledge Base — Phase 1 and Phase 2 ground decisions here. Prevents RESOLVE-IN-PLAN items from recurring story-to-story
id_prefix
Domain Guard key — Phase 0 validates that every Epic ID starts with this prefix. Passing a LAP-Epic ID to the MM pipeline hard-stops with an error before touching any file
services
Phase 2 only — @mm-codebase-planner looks for SERVICE-KB.md in each directory listed here. Services not listed are ignored; services without SERVICE-KB.md fall back to grep
Phase 0 — how domain selection works at runtime
If domain-map.json has one key → auto-selected, no prompt.
If it has multiple keys → numbered menu:
Which team are you working on?

[1] Money Movement        (MM)
[2] Loan Against Property (LAP)
[3] Personal Loans        (PL)

→ Each option auto-loads that domain's
  personas, epics, KB, and services
If the user passes an Epic ID directly (e.g. LAP-Epic-3), Phase 0 auto-selects the matching domain from id_prefix — no menu needed.
Domain Guard — cross-domain hard stop
Fires immediately if the Epic ID prefix doesn't match the selected domain's id_prefix. No files are read. No branches are created. Nothing changes.
❌ WRONG SKILL — Domain mismatch

You passed:  LAP-Epic-3
This skill:  MM pipeline (id_prefix: MM-Epic)

No files read.
No branches created.
Nothing changed.

→ Prevents MM PLAN.md from being
  generated for a LAP epic and vice versa

⚑
Phase 4 · Impact Check
Function Impact Flag — Core Service Logic Change
Triggered only when an existing function body is modified — not when new functions are added. A body change to a shared core function silently breaks every caller in other domain services. This is the most dangerous category of change in a microservice architecture.
Example from this session
Function: DocumentClientService.getPresignedUrl()
Change type: New method added (PARTIAL-REUSE) — no body changes to existing methods
Callers found: Only CustomerPhotoService (new caller) · existing callers untouched
✓ No flag raised — adding a new method does not affect existing callers
When the flag WOULD fire
Trigger
Existing function body in a Core/Shared service is modified — e.g. DynamoService.makeDynamoDBRequest() body changes
Agent traces
Greps all domain services for every call to the changed function — finds every .makeDynamoDBRequest() invocation across customer-service, application-service, artifact-service etc.
Flag shows
Function name · Core service · List of all calling services at risk · Nature of the logic change
Blocks
Implementation proceeds only after engineer reviews each calling service and confirms no breakage — or chooses to add a new function instead
⚑ Impact flag PARTIAL-REUSE only @mm-implementer Silent-break prevention

What changes per domain (in config)
id_prefix Domain guard key — hard-stops any cross-domain Epic ID at Phase 0 before touching a single file
personas Different team = different reviewers, Tech Lead, VP, QA handles — every gate auto-assigns the right people
kb_dir LAP Knowledge Base has LAP-specific field names, status enums, and business rules — Phase 1 and 2 ground in the right KB
services Phase 2 reads SERVICE-KB.md only from this list — irrelevant services are never touched, even if shared repos exist
What stays identical (zero changes)
✓ All 6 subagents: scoping-analyst, codebase-planner, test-architect, implementer, qa-gatekeeper, release-herald

✓ TDD Red → Green pattern, Function Impact Flag logic, p95 latency gate, 8-point PR sign-off
