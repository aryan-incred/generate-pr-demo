Phase 4 · Impact Check
Function Impact Flag — Core Service Logic Change
Triggered only when an existing function body is modified — not when new functions are added. A body change to a shared core function silently breaks every caller in other domain services. This is the most dangerous category of change in a microservice architecture.
Example from this session
Function: DocumentClientService.getPresignedUrl()
Change type: New method added (PARTIAL-REUSE) — no body changes to existing methods
Callers found: Only CustomerPhotoService (new caller) · existing callers untouched
✓ No flag raised — adding a new method does not affect existing callers
When the flag WOULD fire
Trigger
Existing function body in a Core/Shared service is modified — e.g. DynamoService.makeDynamoDBRequest() body changes
Agent traces
Greps all domain services for every call to the changed function — finds every .makeDynamoDBRequest() invocation across customer-service, application-service, artifact-service etc.
Flag shows
Function name · Core service · List of all calling services at risk · Nature of the logic change
Blocks
Implementation proceeds only after engineer reviews each calling service and confirms no breakage — or chooses to add a new function instead
⚑ Impact flag PARTIAL-REUSE only @mm-implementer Silent-break prevention

Command vs Skill — and why /generate-pr is a command
How Claude Code resolves commands/ vs skills/
▼
.claude/commands/ — slash command
📂 Directory can bundle sibling files (e.g. domain-map.json lives alongside SKILL.md)
🔑 Loaded directly into session context when user types /command-name
🎯 Explicit invocation only — never auto-triggered
📋 Becomes the session — it IS the primary instruction context, not a step within one
.claude/skills/ — skill tool
📄 Single file — only the SKILL.md (no sibling config files)
🔧 Invoked via Skill tool — appears in <available-skills> in system context
⚡ Can auto-trigger — trigger conditions in frontmatter let Claude invoke contextually
🧩 Single-phase — does one thing and exits; result returned to caller
Why /generate-pr specifically chose commands/
①
Bundles domain-map.json. The command directory contains both SKILL.md and domain-map.json. This is only possible in commands/ — a skill can only have its SKILL.md.
②
It IS the session orchestrator. Skills are participants in a pipeline. /generate-pr spans Phases 0–7, holds state across phases, resumes on re-run. Being a command means it loads as the primary instruction context — everything else runs inside it.
③
Must load before domain context exists. Skills auto-trigger from conversation context. But Phase 0 reads domain-map.json to establish which domain is active — so the orchestrator must load first, before any domain context exists. Explicit command invocation guarantees this order.

Which service calls which — based on actual SERVICE-KB.md files
▼
Caller → Dependency dynamo-svc document-service hermes-svc lms-connector rules-svc External vendors
paymentservice ✓ ✓ ✓ ✓ ✓ ✓ EaseBuzz · LotusPay · BillDesk · IDFC · Setu
customer-service ✓ ✓ (MM-Epic-1) — — — —
application-service ✓ — — — — —
artifact-service ✓ ⚠️ dead config — — — —
dynamo-svc — ✓ — — — AWS DynamoDB · MySQL LMS/MDM · Redshift · ES · Redis
dynamo-svc — cross-cutting infrastructure
Every service routes ALL DynamoDB operations through dynamo-svc. It's not a domain service — it's infrastructure. Also proxies MySQL (LMS, MDM, Reconciliation), Elasticsearch, and Redshift. Any body change to its core functions triggers Function Impact Flag across all callers.
Shared services — appear in multiple domain configs
customer-service and application-service are listed in domain-map.json for both MM and LAP teams. SERVICE-KB.md is read independently per pipeline — no conflict. A body change to a shared function in either service would trigger Function Impact Flag for both domains.
External vendor isolation
EaseBuzz, LotusPay, BillDesk, IDFC First Bank, Setu/BBPS — all called only from paymentservice. Intentionally isolated. No other domain service can call an external payment vendor directly.
⚠️ artifact-service — dead document-service config
documentServiceUrl referenced in k8s config for artifact-service, but zero client code in the service. Config key exists, no HTTP calls made. Future integration, not yet implemented.

Why sonnet-4-6[1m] for P2 and P4?
P2 Blueprint — @mm-codebase-planner reads all SERVICE-KB.md files for every service in scope, plus domain-map.json, plus the story ACs. A 5-service story can easily put 50–80k tokens into context before writing a single line of PLAN.md. Standard window risks truncation.

P4 Implementation — @mm-implementer holds PLAN.md + all failing test file contents + modified source files in context across every chunk. Multi-chunk stories accumulate context; without 1M window, later chunks lose earlier context and introduce regressions.
Why haiku for P0, P5, P6, P7?
These phases are mechanical — they execute shell commands, compare checklist items, fill templates, or call GitHub APIs. No multi-step reasoning is required. Haiku handles them correctly at ~5× lower cost than Sonnet, and these phases combined typically use <20% of total story tokens.
